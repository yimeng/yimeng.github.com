总结

写在前面的

工作5y+了,被人要起来东西的时候才想到开始总结.开始拿脑袋想了下.写过不少好玩的好用的.但是基本上也没有一个总结和积累.比较可惜.那些过往的代码,可能的地方估计有两个:

  • 和某些小电影放到了以前的移动硬盘(NAS)里.
  • github上之后很少很少的东西.

鉴于我比较懒的缘故实在去翻那些代码.所幸就在下面自己总结吧.也算完成某人给的任务==

监控

大学时代

当时闲着没事,在网上找了一篇mrtg的文章.第一个练手的就是学校访问量最大的网站(42)的服务器.依稀记得当时是对照教程把服务器上的SNMP打开了.记忆里应该是没有任何的SNMP的安全措施.还好应该是只读的.

当然直接在生产服务器上做实验的这事.当时还小.不懂事:)这个在东软时期有过严重的教训.这个又可以开一个专题了.

成就:发现了首页的一个巨大的FLASH占用了大量的带宽,把建议提给了某同学,却被说浏览器应该有缓存.妈蛋.我这流量咋解释.虽然没改变什么,但好歹从监控发现了一些可以优化的问题.成就感还是有的.

东软时期

背景

开始正式的系统管理员的生活.当时碰到了一个最最经典的情景:写监控系统的不收报警.收报警的无法改监控系统.

在以后5年多的SA生涯里,这个模型依旧存在,即使DevOps的兴起也没有改变什么.

既然不能利用自己的一线工作经验去调教监控系统.只能自力更生了.

调研

花了一个月的时间开始调研以下的监控工具:

  • nagios
  • zabbix
  • centreon
  • bigbrother
  • zenoss
  • opennms
  • pandora

现在看看,除了前两个,其它的应该没人知道了.当时大家对于zabbix应该是刚刚开始知道.网上只有零零散散的几个介绍文章.因此当时主要还是研究了下nagios.

选型

优势
  • 灵活,灵活,灵活.
  • 监控的调度做的非常好.
  • 报警很完善.自定义时间段,升级.
  • 事件钩子机制
劣势
  • 没有图,没有图,没有图
  • 一些插件还需要自己写(社区刚刚起步)
  • 插件对Solaris的支持.

插件自己写这事可以忍,但没有图这事咋办?当时在nagios群里有人推荐了centreon.当时也是听说淘宝那边也在用这个.

搭建centreon

按照官方的方式,还要安装一堆依赖.当年要是有docker该多好(其实有zone了).当时第一次感觉.官方的教程如此得靠谱.

编写nagios插件.

使用Shell,模拟用户真实发送邮件.要模拟内网各种部门的邮件互通(落地在不同的服务器),还要模拟内外网的邮件互通(走MX外网邮件网关),因此逻辑关系会有7*7这么多.体力活. 具体使用的应该社区上的一个现成的插件进行了修改.具体的细节就忘记了==

监控系统

东软

根据 nagios 的插件标准使用 msmtp命令模拟用户发送邮件行为,发送邮件附带相关 request id 信息,接收端通过 pop 协议收取邮件.通过判断 request id 实现一个发送邮件和接收邮件的闭环监控.由于nagios 没有图表展示,根据当时淘宝的技术分享,使用了 centreon 集成方案作为最后方案.由于监控项和服务项比较少,所以没有碰到相关的性能问题.当然,顺便也用 NRPE 实现了磁盘 CPU 等基础监控.

新浪微博

监控

使用 centreon 通过 SNMP 协议监控网络流量,依旧使用 NRPE 去监控CPU 内存等资源.外加端口监控. 监控规模扩大到了600台.

配置管理

由于当时是 CentOS 5 python 版本是2.4 , yum 对 python 还有着强依赖.又没有现在的virtualenv.虽然当时有fabric.但由于 pyhont 版本要求,并没有使用. 最后选型使用 puppet, 但由于当时的架构师puppet 的版本绑定到数据库中.丧失了 puppet 原有的灵活性.最后大家只使用了一个 puppet 的 exec 执行了一个 shell 脚本.一直沿用了3年.

##介绍 puppet

什么是 puppet

  • 配置管理工具
  • 系统自动化的框架
  • 一种描述语言
  • 开源软件
  • 跨平台
  • 好多人用
  • 巴拉巴拉

配置管理的优势

  • 基础设置即代码

    追踪 测试 部署 扩容 代码变更都可以用软件工程那套配置管理啦

  • 幂等性

    每次运行的结果都一致

  • 快速扩容

    扩容1台机器和100台机器是一样滴(现实并非如此)

  • 一致的用户体验
  • 统一环境

    测试 生产 开发
    类似软件 Chef CFEngine Salt Ansible

puppet 版本介绍

版本号介绍:

  1. 主版本号：当你做了不兼容的API 修改，
  2. 次版本号：当你做了向下兼容的功能性新增，
  3. 修订号：当你做了向下兼容的问题修正。
  • 0.9.x - 第一个测试版本
  • 0.10.x - 0.21.x - 各色各样的早期版本
  • 0.22.x - 0.25.x - 改进并且支持各种操作系统版本
  • 2.6.x - 许多重要的更改,参数类(见识了某系统恶心的设计)
  • 2.7.x - 终于开始支持 windows 了
  • 3.0.x - 许多更改,禁止了动态变量范围,数据绑定.
  • 3.2.x - 为升级到4做了一些准备,比如环境目录
  • 4.x.x - 2015年发布,新的语法分析,有了许多更改和一些向后不兼容的鸟毛病,把 master 做成了一个 all in one 的怪物

###brew
使用方法啥的估计一搜一大堆,这里不多说了.这里说说一些注意事项:

关于图床的选择

第一篇正经的文章居然是:关于博客床的选择.在大学的时候就有这种纠结.

dropbox

要翻x,这个就不多说了.应该是用dropbox的外链功能.风险较高.

七牛

需要上传个人拿着自己的身份证的照片认证.才能升级为标准用户.才可以有免费10G的空间和流量.
可以自定义域名,比较新.网上负面的评价比较少.看起来还是比较靠谱的.

upyun

需要在网站低下加一个logo连接才可以享受10G空间15G流量.懒得弄了.暂时还没有尝试.

##又拍(Yupoo)
老牌的图片托管了,记得我上大学的时候就开始用.对于个人用户还是比较友好的.但是总是感觉用不惯.不知道是界面的缘故还是什么.可能原来只是为了存我自己的照片.个人感觉那就是国内的Flickr.不过后来的又拍云才更像图片托管的概念吧.

##微博图床
呃,这个,临时有个图片分享啥的用用还是不错的.对于图床的部门.呃.了解不太多.不过可能为了微博.这点”浪费”渣浪可能感觉还是可以忍受的.神马CDN都不用操心.so还是量大实惠的.适合对于图片安全性要求不高,但是对于速度有要求的同学.

##Flickr
大学的时候曾经想过一直用它.可惜.后来被雅虎收购,我的ID一直找不到自己喜欢的.当时墙反而不是考虑的因素了.没错就是没有自己喜欢ID才放弃了它.

##google
高中的时候开始用,前一阵子突然发现了一个高二时候的照片.可惜当时没多传几张上去.10年了.那个照片居然还在.可见还是很坚挺的.不过按照最近几年google的路子.没到不会没,就是可能会向SNS方向走了.忘记看有没有外链了.不过google在国内这样.有没有都可以略过了==

##总结
博客暂时体会下七牛,备选upyun.当然要用自己的域名. 不太重要的就微博图床了.对了这里推荐一个软件:dropzone,上传图片啥的还是很方便的.60大洋值得拥有:)
